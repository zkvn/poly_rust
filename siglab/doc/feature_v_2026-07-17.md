# Feature: run the V-shape grid (incl. `v_0.7_0.3_0.7`) on weather/World Cup buckets too

Status: **proposed — pending DeepSeek review before implementation.**

## What's not supported today

`v_0.7_0.3_0.7` is the threshold-triple prefix of `v_shape.rs`'s 16-variant grid (full IDs
like `v_0.7_0.3_0.7_0.3_0.05` — see `TRIPLES` in `v_shape_grid()`). It already runs, and has
run since 2026-07-15/16, but **only against crypto markets**:

- `market.rs::run_market` builds one `VShapeEngine` per `(crypto market, grid variant)` pair,
  unconditionally, for every 5m/15m/4h/hourly-ET market.
- `event_monitor.rs::run_event_feed` (shared by `weather.rs`/`worldcup.rs`) only builds
  `BucketReversalEngine`s from `bucket_reversal::reversal_grid()` — no `VShapeEngine` at all.
- `record.rs::SiglabTradeRecord::from_v_shape_engine` hardcodes `market_kind:
  MarketKind::Crypto`, confirming this isn't an oversight in one call site — the type itself
  assumes V-shape is crypto-only.
- `report.rs`'s generated "Strategy config" text says so explicitly: *"Crypto markets
  additionally run the 16-variant V-shape grid... weather/World Cup buckets [trade only]
  the same `reversal_{low}_{high}` 18-combo grid."*

So today, a weather city or World Cup bucket never gets a chance to fire a V-shape trade —
this is the gap the user asked to close.

## Why this is a reasonable extension (not a hack)

`VShapeEngine` doesn't actually need a crypto cycle to function correctly:

- Its only cycle-aware behavior is the force-unwind-within-10s-of-`cycle_end_ts` branch in
  `on_tick`, gated by `if self.cycle_end_ts > 0.0 && ...`.
- `VShapeEngine::new` initializes `cycle_end_ts: 0.0`, and that field is only ever mutated by
  `cycle_open()`.
- If a caller simply **never calls `cycle_open`**, that branch is permanently unreachable
  (`0.0 > 0.0` is always false), leaving stop-loss / take-profit / `UNWIND_TIME_SECS` (25s)
  timeout as the only exit paths — which is *exactly* `bucket_reversal.rs`'s own exit model
  (no cycle, no resolution, SL/TP/timeout only; see that file's doc comment and
  `doc/plan_weather_worldcup_trading_2026-07-13.md`).

So wiring `VShapeEngine` into `event_monitor.rs` the same way `BucketReversalEngine` already
is wired requires **zero changes to `v_shape.rs` itself** — the engine already degrades
correctly to a no-cycle mode by construction.

## Design

1. **`record.rs`** — generalize `SiglabTradeRecord::from_v_shape_engine` to take a
   `market_kind: MarketKind` parameter, the same way `from_bucket_engine` already does,
   instead of hardcoding `MarketKind::Crypto`. Update its doc comment (currently says "V-shape
   only runs on crypto markets, unlike bucket_reversal"). Update `market.rs`'s two call sites
   to pass `MarketKind::Crypto` explicitly — no behavior change there.

2. **`event_monitor.rs::run_event_feed`** — alongside the existing
   `HashMap<U256, Vec<BucketReversalEngine>>`, build a second
   `HashMap<U256, Vec<VShapeEngine>>` from `crate::v_shape::v_shape_grid()`, one set per
   bucket, constructed the same way (never calling `cycle_open` — see above). On each
   incoming tick, feed it to both engine maps (same `up` value, same `ts_secs`); any
   `ClosedTrade` from the V-shape engines is forwarded through the same `trade_tx` via
   `SiglabTradeRecord::from_v_shape_engine(..., market_kind, ...)`, mirroring the existing
   `from_bucket_engine` forwarding right below it.

3. **`report.rs`** — update the generated "Strategy config" paragraph (~line 720-733) to
   state that weather/World Cup buckets now also run the V-shape grid, not just the reversal
   grid. The existing "Crypto V-shape variants" `<details>` table (~line 682) either gets
   renamed (it's no longer crypto-only) or gets a one-line note added — leaning toward a
   rename + note, decide during implementation once the table's actual layout is in front of
   me.

4. **`main.rs`** module doc comment — currently says v_shape is "per (crypto market, grid
   variant) pair" only; update to reflect it also runs per (weather/World Cup bucket, grid
   variant) pair.

## Resource-scale note

Weather/World Cup buckets currently run 18 `BucketReversalEngine`s each
(~51 cities × ~11 buckets + ~62 events × up to 60 buckets — tens of thousands of instances
total, per `doc/plan_weather_worldcup_trading_2026-07-13.md`'s own scale note, verified cheap
empirically after that feature shipped). Adding the 16-variant V-shape grid is roughly a
+89% increase in engine count for these markets specifically (34 engines/bucket instead of
18). Each `VShapeEngine::on_tick` call is a few floats and branches, same cost profile as
`BucketReversalEngine::on_tick` — expected to stay cheap, but this needs the same kind of
empirical check (Docker resource usage, not just an assumption) before calling it done, same
discipline as the original bucket_reversal rollout.

## Testing plan

- `v_shape.rs`'s existing unit tests are untouched and stay green (crypto path, `cycle_open`
  still called there, unaffected).
- New tests around the event-feed wiring (exact location depends on how testable
  `run_event_feed`'s internals are without a live WS connection — likely a smaller
  unit test at the `VShapeEngine`-in-no-cycle-mode level, analogous to
  `v_shape.rs`'s own tests but asserting cycle_end_ts is never set / force-unwind branch never
  fires) verifying:
  - A `VShapeEngine` that never receives `cycle_open` still fires entries/exits correctly via
    SL/TP/timeout only.
  - `SiglabTradeRecord::from_v_shape_engine(MarketKind::Weather, ...)` /
    `MarketKind::Worldcup` round-trip correctly (matches existing `from_bucket_engine`
    round-trip test pattern already in `report.rs`, if one exists, or a new one in
    `record.rs`).
- Full suite clean: `cargo test`, `cargo fmt --all --check`,
  `cargo clippy --all-targets --all-features -- -D warnings`.
- **Local Docker verification only, no Oracle deploy** (per explicit request): build and run
  `siglab`'s own `docker-compose.yml` locally, watch logs for weather/World Cup V-shape engine
  startup + at least one real closed trade landing in the trade log / hourly report with
  `strategy: "v_shape"` and `market_kind` of `weather` or `worldcup`, and spot-check
  CPU/memory didn't regress meaningfully from the pre-change baseline in
  `doc/local_resource_test_2026-07-13.md`.

## Non-goals

- No change to crypto's existing V-shape behavior or grid.
- No real-outcome / Gamma resolution — same philosophy as `bucket_reversal.rs` throughout.
- Not deploying to Oracle — this is a local-Docker, paper-trading-only verification per
  explicit request.

## Open questions for DeepSeek's review

1. Is reusing the *exact same* 16-variant grid (same `high1`/`low`/`high2`/`sl_pnl`/
   `unwind_pnl` values tuned loosely off crypto's 5-minute cycles) reasonable for weather/
   World Cup's much thinner, slower order books, or should this ship with a caveat (like
   `bucket_reversal.rs` already carries) rather than a claim of tuned parameters?
2. Any correctness concern with feeding the *same* engine construction path (`v_shape_grid()`)
   into two very differently-behaved market classes (crypto's fast-moving cycles vs.
   weather/World Cup's rare-tick buckets) purely by skipping `cycle_open`, versus something
   more explicit like a constructor flag?
3. Given weather/World Cup buckets already run 18 reversal variants, does adding another 16
   (34 total/bucket) still fit the "expected cheap" assumption, or does this warrant a
   pre-implementation estimate rather than a post-hoc check?
