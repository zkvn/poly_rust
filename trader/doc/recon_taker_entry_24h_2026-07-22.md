## Every trade, in order

`p(side)` is the model's probability of the side actually traded (p(up) for a
UP entry, 1−p(up) for a DOWN entry) — the same number the entry gate itself
checks against. `Edge` = `p(side) − price` at that moment: positive means the
model thought the side was underpriced, negative means it disagreed with the
entry price. These p(side) readings are pulled from the nearest console
heartbeat line in `live.log` before each timestamp (heartbeats print roughly
every 30s per asset — there's no per-trade indicator column persisted
anywhere), so treat them as accurate to within ~30-60s, not to the exact
millisecond of the fill.

## Live vs backtest (BT) cross-check

The same window was independently replayed through the Rust `backtest`
engine against the same price data and same config, then compared trade-by-
trade against what actually happened live. This checks whether the paper
engine's real-time execution matches what the offline replay says *should*
have happened — a second, independent check on top of the win-rate table
above.

**5 of 12 matched exactly** (same outcome, pnl within a few cents — the small
gap is expected timing/pricing noise between a live tick-by-tick fill and an
offline replay, not a bug). Of the remaining 7:

- **4 outcome differences** (BNB×3, ETH×1) — live closed differently (WIN
  instead of UNWIND, or vice versa) than the backtest replay would have. All
  four are explained the same way: live was inside a **drawdown/halt window**
  at that point (a real safety mechanism — see `render_bt_reconciliation`'s
  `Reason` column) that the unconstrained backtest replay doesn't apply. Not
  a live-vs-backtest logic bug — a live-only safety behavior working as
  designed.
- **1 side difference** (DOGE 05:44:45) — live and BT disagreed on which side
  even should have entered, again attributed to a live halt window at that
  moment.
- **2 "BT did not fire"** (both DOGE) — live traded during a window the
  backtest replay considers halted, so it never generated a comparison row at
  all. Same halt-window cause as above.

**Backtest vs live (the reverse check — cycles the backtest would have
traded that live never touched): zero.** Every cycle the backtest fired,
live also traded — no missed opportunities, no silent gaps in live's own
entry logic.

**Bottom line:** every mismatch in this window traces to live's own halt/
drawdown safety windows suppressing or altering an entry the unconstrained
backtest replay doesn't know about — not to a discrepancy in the core trading
logic itself. Full row-by-row detail (entry Δ%, cycle Δ%, exact reason
classification) is in the technical report linked at the top.
