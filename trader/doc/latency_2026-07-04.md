# Latency study — 2026-07-04

Scope: (1) how delayed the CLOB price feed is by the time the Rust bot sees it, how long
signal-processing + order placement + Polymarket's own confirmation take (entry and exit legs
separately), and (2) how the current NATS-based infrastructure actually handles WS subscriptions —
full firehose vs. filtered, and whether any internal polling/sampling silently drops messages.

**Bottom line up front:**
- The CLOB price feed itself is fast and well-behaved: **p50 ≈ 4–5ms, p95 ≈ 15–17ms** from
  Polymarket's own timestamp to the Oracle box receiving it, for every asset. This is not the
  bottleneck.
- The Binance feed carries a **constant ~113–116ms floor** (real network distance to Binance,
  not jitter — p50/p95/p99 are nearly identical), plus it's **explicitly sampled/coalesced at
  250ms**, so bursts of Binance trades faster than 4/s are silently thinned before the trader or
  the parquet record ever sees them.
- **There is currently zero timing instrumentation anywhere in the order-placement path.** No
  code in `trader/src/execution.rs` records when an order was submitted, when Polymarket's
  response came back, or how long the round trip through the EC2 proxy took — for either the
  entry BUY, the stop-loss/cycle-end market SELL, or the unwind GTC limit SELL. This is the
  single biggest blind spot and directly blocks answering "how long does Polymarket take to
  confirm an order" from existing data. §4/§5 below.
- WS subscriptions are **not** a full un-filtered firehose — the SDK subscribes by explicit
  asset-ID list at the protocol level and filters server-and-client-side by asset_id before
  handing messages to each consumer (§6.1). But there are two real "an update went missing"
  mechanisms already in production, both currently invisible in the logs: the SDK's internal
  broadcast channel can silently drop messages under backpressure (`RecvError::Lagged`) and
  that warning is compiled out because the `tracing` feature isn't enabled (§6.2); and the
  Binance feed (and the parquet-recorded copy of the Poly feed) is only sampled every 250ms/
  200ms, so genuine intermediate ticks are overwritten before being persisted (§6.3). The live
  trader's own Poly-price signal (via NATS) is *not* subject to that second issue — it publishes
  every merged best-bid-ask/price-change event immediately, uncapped.

## 2. Data path (what actually runs where)

```
Polymarket CLOB WS  ──(internet)──>  Oracle box (10.8.0.1)
                                        │  price_feed collector
                                        │  subscribe_orderbook / subscribe_best_bid_ask /
                                        │  subscribe_prices / subscribe_last_trade_price
                                        │  (all on ONE shared "Market" WS connection, §6.1)
                                        │
                                        ├─> parquet (raw/raw_15_mins/raw_4hr) — 200ms/250ms/1s
                                        │   sampled snapshot, see §6.3
                                        │
                                        └─> local NATS (127.0.0.1:4222, loopback only)
                                              │
                                              ▼
                                        trader (live driver, SAME box)
                                        subscribes price.poly.<ASSET> / price.binance.<ASSET>
                                        worker.step() — pure sync fn, no I/O (§4)
                                              │
                                              ▼ (order intent)
                                        LiveExecutionEngine::place / close_position /
                                        place_limit_sell  →  HTTPS via EC2 proxy (10.8.0.7:8888)
                                        →  clob.polymarket.com  (§5)
```

Both the collector and the trader run on the **same** Oracle box and talk over **loopback**
NATS — so the NATS hop itself contributes negligible latency (not measured here; it's
sub-millisecond in any local-loopback pub/sub and isn't the bottleneck in this pipeline). The
only place a *second* real network hop exists is order writes, which are proxied through EC2 to
work around a geoblock (`CLOB_PROXY_URL` / `HTTPS_PROXY`, see `trader/src/bin/live.rs:580-587`).
That means every order-placement HTTP call pays Oracle→EC2→Polymarket, not just Oracle→Polymarket
— and that's exactly the leg with no timing data (§5).

The Binance number is a **flat ~115ms floor**, not jitter (p50 ≈ p99 for every asset) — this is
real network distance from Oracle to Binance's infra, essentially constant. Compare to Poly's
single-digit-ms floor: either Polymarket's CLOB WS edge is materially closer to Oracle (same
region / same cloud peering), or Polymarket stamps `server_ts` at a point in its stack closer to
the wire than Binance's `E` (event-time) field does. Either way, **Binance is the slower feed by
~20x on typical latency**, which matters because entry signals in this bot are Binance-price-
triggered (`bot/worker.py` / `worker.rs`'s reversal/high_prob logic watches Binance deltas) — the
~115ms Binance floor is baked into every signal's reaction time, on top of whatever the 250ms
sampler adds (§6.3).

## 4. Signal-processing latency (in-process)

`Worker::step()` (`trader/src/worker.rs:438`) is a **plain synchronous function**, not `async` —
it does no I/O, no locking beyond what's already resolved by the time it's called, and returns a
`Vec<Action>` for the driver to execute. Tick → decision inside this function is pure CPU work
(state-machine transition + arithmetic), on the order of microseconds, not something that shows
up next to network-bound legs. There's no code path here that could plausibly be the bottleneck,
and there's no existing instrumentation to prove that quantitatively — it's an argument from
reading the code, not a measurement. If it's ever worth measuring precisely (e.g. to rule it out
formally in an incident writeup), wrap the `worker.step()` call sites in `live.rs` with
`std::time::Instant` and log the delta; this is cheap to add (§8).

## 5. Order placement → confirmation latency — not instrumented (gap)

Traced every order code path in `trader/src/execution.rs`:

- **Entry (FAK BUY)** — `LiveExecutionEngine::place()` (execution.rs:279-353): one
  `.build_sign_and_post(&self.signer).await` per attempt. FAK orders resolve synchronously in the
  HTTP response itself (fill-or-kill semantics — the response *is* the confirmation), so "how
  long does Polymarket take to confirm an order" for entries is really "how long does this one
  HTTPS round trip (via the EC2 proxy) take."
- **Exit market SELL (stop-loss / cycle-end)** — `close_position()` (execution.rs:412-474): same
  pattern, one HTTP round trip per attempt, up to `close_max_retries` (5) attempts.
- **Exit GTC limit SELL (unwind take-profit)** — `place_limit_sell()` (execution.rs:355-410): one
  HTTP round trip to place; if it doesn't fill immediately it rests on the book as `SellStatus::Live`
  with no separate fill-confirmation timing captured anywhere.

**None of these three call sites wrap the network call in `Instant::now()` / log a duration.**
Grepped the whole crate for `Instant::now` (execution.rs and unwind.rs: zero hits) — the only
`Instant::now()` usage in the entire trader binary is `balance_deadline` scheduling in
`live.rs:781/985`, unrelated to order timing.

Compounding this, `trader/src/unwind.rs`'s `UnwindWatcher` — a real-time USER-channel fill-push
listener, which is exactly the mechanism that *would* give an accurate, event-driven "order
submitted → exchange confirmed fill" timestamp pair for the unwind leg — **is fully implemented,
unit-tested, and never invoked from `live.rs`** (confirmed by grep: `UnwindWatcher` only appears
inside `unwind.rs` itself). The doc comment on `UnwindWatcher::run` says as much: "NOT invoked
anywhere yet — requires real Polymarket API credentials and is part of the same B2 go-ahead gate
as live order placement." The live driver currently only ever finds out about a resting
GTC-sell's fill by re-deriving state from the next `PlaceLimitSell`/position-check cycle, not by
listening to the USER channel.

Also checked whether the console log (`trader/live_logs/live.log`) could substitute for explicit
instrumentation via line-arrival timestamps (e.g. via `journalctl`'s per-line reception time).
It can't: per the README ("live trader (systemd — tails the same live.log StandardOutput is
appended to")), stdout is redirected straight to a plain file, not captured through the systemd
journal — so there's no per-line timestamp metadata at all, and the `println!`/`eprintln!` calls
in `execute()` (`live.rs:355-448`) don't embed a timestamp themselves either (only the *Telegram
notification* text does, via `hkt_now().format("%H:%M:%S")`, which is second-resolution and only
sent after the whole action already completed).

**Net: no data exists today, from parquet or from logs, to answer "how long does Polymarket take
to confirm an entry/exit order" or "how long does the EC2 proxy hop add."** This is a hole, not a
finding — see §8 for what to add.

## 6. WS subscription handling — firehose vs. filtered vs. polled

### 6.1 Not a firehose — explicit per-asset subscription, filtered at two layers

Every subscription call in `price_feed/src/collect.rs` and `trader/src/marketdata.rs` passes an
explicit `Vec<U256>` of asset IDs (`subscribe_orderbook(ids)`, `subscribe_best_bid_ask(up_ids)`,
`subscribe_prices(up_ids)`, `subscribe_last_trade_price(ids)`) — never a bare "subscribe to
everything" call. Reading the SDK source
(`polymarket_client_sdk_v2-0.6.0-canary.1/src/clob/ws/subscription.rs`):

- `SubscriptionManager::subscribe_market_with_options` sends a `SubscriptionRequest::market(...)`
  naming only the *newly*-requested asset IDs (ref-counted — re-subscribing to an already-covered
  asset sends nothing new, `subscription.rs:243-263`), so the wire-level ask to Polymarket really
  is scoped to the assets in play, not the whole exchange.
- All four `subscribe_*` calls multiplex over **one shared physical "Market" WS connection**
  (`get_or_create_channel(ChannelType::Market)`), reused across the 5m/15m/4hr durations that
  `price_feed::collect::run()` spins up — so this is one WS connection to Polymarket total for
  all book/bba/price/trade data across every asset and every duration, not one per subscription.
- Each `subscribe_*` call's returned stream is filtered **inside the SDK itself** by asset_id
  before yielding (`subscription.rs:280-306`: `asset_ids_set.contains(&book.asset_id)` etc.) — so
  the app-level re-filtering already present in `collect.rs`/`marketdata.rs` (with the comment
  "always filter by asset_id explicitly... matching price_feed's proven pattern") is *defensive
  redundancy*, not compensating for a real SDK gap. Harmless, but worth knowing it's belt-and-
  suspenders rather than load-bearing.

### 6.2 Real silent-drop path: broadcast backpressure, currently unobservable

The SDK's per-subscription stream reads from a `tokio::sync::broadcast` channel
(`subscription.rs:281`: `rx.recv().await` matched against `RecvError::Lagged(n)` /
`RecvError::Closed`). A `broadcast` channel has a bounded ring buffer — if any one subscriber
(e.g. the 4hr book task, which processes messages far less urgently than the 5m task) falls
behind the shared connection's send rate, the channel drops the oldest unread messages for that
lagging subscriber and returns `Lagged(n)` on its next `recv()`, i.e. **an explicit, countable
"we silently missed n messages" signal already exists in the SDK.**

But: `subscription.rs:314-317` only surfaces this via
`#[cfg(feature = "tracing")] tracing::warn!(...)`, and neither `trader/Cargo.toml` nor
`price_feed/Cargo.toml` enables the SDK's `tracing` feature (`features = ["ws", "clob", "rtds"]`
in both — confirmed by grep). So today, **every `Lagged(n)` event is silently compiled away** —
the `#[cfg(not(feature = "tracing"))] let _ = n;` branch runs instead and the drop count is
discarded with zero log output. There is currently no way to tell, from any log this system
produces, whether a burst of book/price/trade updates ever outran a subscriber. Cheap, high-value
fix — see §8.

### 6.3 Real coalescing path: fixed-interval samplers (this one is deliberate, not a bug)

`price_feed::collect::run()` (collect.rs:1233-1296) drives three tickers — `200ms` (poly/book,
5m+15m), `250ms` (Binance), `1s` (poly/book, 4hr) — each of which takes whatever the shared
per-asset `Mutex<AssetState>`/`Mutex<BinanceState>` currently holds and writes/publishes *that*
snapshot. Any WS update that lands and is then overwritten by a newer one before the next tick is
never persisted or published from this path — this is intentional (matches the recording
granularity the strategies need), but it does mean the parquet-recorded tick count under-samples
the true WS message rate. Measured via inter-arrival gaps on `server_ts` (2026-07-03, `raw/`):

| Feed | Asset | median gap | p90 gap | %gaps ≤ 210ms (sampler period) |
|---|---|---:|---:|---:|
| Poly | BTC  | 200ms | 209ms  | 91.9% |
| Poly | ETH  | 200ms | 233ms  | 74.8% |
| Poly | BNB  | 217ms | 478ms  | 46.6% |
| Poly | HYPE | 220ms | 567ms  | 45.8% |
| Binance | BTC  | 450ms  | 1428ms | 16.2% |
| Binance | ETH  | 404ms  | 1565ms | 17.8% |
| Binance | DOGE | 1375ms | 6459ms |  7.8% |

Two different mechanisms sit inside these numbers:

1. **The parquet copy of the Poly feed is under-sampled** (it's a 200ms snapshot of a faster
   feed) — for a busy asset like BTC the true update rate is *close to* the 200ms sampler rate
   already (92% of gaps are right at the floor), so relatively little is actually lost; for a
   quieter asset like HYPE the underlying feed itself may just update less often, so the gap
   number partly reflects genuine market quietness, not sampler loss — can't fully separate the
   two without an unsampled capture to diff against (§8 proposes exactly that).
2. **Critically, this sampling limitation does *not* apply to what the live trader itself acts
   on.** `spawn_bba_task` (collect.rs:940-1051) publishes to NATS **immediately, per merged
   best-bid-ask/price-change event**, inside the same loop that updates the shared state — it is
   not gated by the 200ms ticker at all. The ticker only governs the *parquet-recording* and
   *Binance-NATS* paths. So the strategy code trading on Poly price sees every event in real
   time; only the historical record of it is thinned.
3. **The Binance feed genuinely is throttled for the trader too**, not just for recording —
   `spawn_binance_task` updates a shared `BinanceState` on every trade, but `ticker_250ms` is the
   *only* place that publishes to `price.binance.<ASSET>` on NATS (collect.rs:1263-1284). At
   7.8–17.8% of gaps landing inside one 250ms window, there's direct numeric evidence that
   multiple genuinely-different Binance trade prices are colliding into a single sample more
   than occasionally for every asset checked — and since entry signals key off Binance deltas,
   this is a real (if modest, ~100-200ms class) added reaction lag on top of the ~115ms network
   floor from §3.2.

### 6.4 One clearly wasted (not lossy) subscription: DOWN-side book updates

`spawn_book_task` (collect.rs:867-933) subscribes to **both** `up_id` and `dn_id` per asset for
the orderbook channel, but `write_sample`'s caller only stores/acts on updates where `is_up` is
true (collect.rs:909-921: `if is_up { ...update state... }` with no `else` branch at all) — every
DOWN-token book message is received off the wire, matched against the asset map, and then
dropped on the floor unconditionally. Not a latency or correctness bug (DN price is always
derived as `1 - UP` elsewhere — `write_sample`, collect.rs:673-674: `dn_bid_p`/`dn_ask_p` are
computed as `1.0 - ask_p`/`1.0 - bid_p`, never taken from the actual DN-token book message), just
unnecessary WS traffic/CPU for a duplicate feed nothing reads.

**Feedback (2026-07-04): upgraded from cleanup to an action — see G in §8.** Rather than drop the
DN-side message, persist it (the actual received DN book, not just the derived `1 - UP` complement)
so the complement assumption can be checked against real data instead of assumed.

## 8. Proposed plan

Ordered cheapest/highest-value first.

**A. Turn on the SDK's existing lag visibility (near-zero effort).**
Add `"tracing"` to the `polymarket_client_sdk_v2` feature list in both `trader/Cargo.toml` and
`price_feed/Cargo.toml`, and call `tracing_subscriber::fmt().init()` (or route into the existing
`println!`-based logging via a custom subscriber layer, to avoid a second log format) at the top
of both `main()`s. This alone makes every `RecvError::Lagged(n)` in the SDK visible for the first
time — directly closes §6.2 with no new code of our own to maintain.

> **Feedback (2026-07-04) — Q: if 5m/15m/4hr each get their own WS subscription, is A still
> needed?** Yes, unchanged. Checked the SDK source
> (`subscription.rs::subscribe_market_with_options`): every `subscribe_*` call already gets its
> **own independent `broadcast` receiver** (`let mut rx = self.connection.subscribe();`) —
> that's true today even though all three durations share one physical connection (`clob =
> ClobWsClient::default()` in `collect.rs::run`, cloned into every book/bba/trade task for 5m,
> 15m, *and* 4hr). Wire traffic is already deduplicated too: `subscribed_assets` is ref-counted,
> so only the *first* duration to subscribe to a given asset actually sends a request over the
> wire; the other two durations' subscriptions are pure in-process receivers layered on the same
> connection. So a slow 4hr consumer already can't steal messages from 5m/15m — that isolation
> exists at the per-subscription-receiver level, not at the connection level. Giving each
> duration its own physical connection would only make Polymarket send the same market data three
> times instead of once (more wire/CPU, no latency benefit) and wouldn't change the Lagged-drop
> mechanism at all, since that's a property of each receiver's own drain rate vs. the shared
> sender, independent of how many physical connections feed it. **A stays on the plan regardless
> of whether that architecture change ever happens.**

**B. Instrument the three order-placement call sites (small, contained change).**
In `execution.rs`, wrap each `.build_sign_and_post(&self.signer).await` (in `place`,
`place_limit_sell`, `close_position`) with `let t0 = std::time::Instant::now(); ...; let rtt_ms =
t0.elapsed().as_millis();` and thread `rtt_ms` back out through `TradeResult`/`CloseResult`/
`LimitSellResult` (one new field each) so `live.rs`'s `[ORDER]` print lines can log it directly.
This closes §5 for the "HTTP round trip via EC2 proxy" leg — the dominant unknown — without
touching the SDK or adding a new service. Also add a wall-clock timestamp (`SystemTime::now()`
ms) to every `[ORDER]`/`[SL]`/`[close]` print line in `live.rs`, since the current log has none at
all (§5) — cheap, and fixes the log-correlation problem generally, not just for orders.

> **Feedback (2026-07-04):** OK as written, no changes.

**C. Time the tick → action gap (closes §4 with real numbers instead of an argument from code
reading).** In the `binance_rx.recv()`/`poly_rx.recv()` arms of `live.rs`'s main select loop,
record `Instant::now()` right after `.recv()` returns and again right after `driver
.process_actions(...).await` returns; log the delta whenever an `Action` other than a no-op was
produced. Confirms the §4 claim quantitatively and would catch a regression (e.g. a future change
that accidentally adds blocking work to `worker.step()`) that pure code review can't.

> **Feedback (2026-07-04):** OK as written, no changes.

**D. Generalize the unwind fill listener into a dedicated order-latency watcher (moderate effort,
real functional upgrade, not just observability).**

> **Feedback (2026-07-04):** rename `UnwindWatcher` → `entry_exit_watcher` and scope it to *all*
> order legs, not just the unwind GTC sell — purpose-build it to answer "how long does
> Polymarket take to confirm an order," end to end.

`trader/src/unwind.rs` already has a tested USER-channel fill listener that's simply never
started, currently scoped only to the resting GTC take-profit sell. Broaden it (rename the
module/struct to `entry_exit_watcher`) to `watch()` the order_id from **all three** legs —
entry FAK BUY, exit market SELL (stop-loss/cycle-end), and exit GTC limit SELL — right after each
is submitted, recording the submit `Instant` (from B) against the USER-channel push confirmation
timestamp. For the FAK entry and market-exit legs this gives a second, independent confirmation
timestamp to cross-check against the HTTP-response timing from B — the HTTP 200 tells you "the
proxy/exchange answered this request," not necessarily "the matching engine confirmed at this
exact instant," so comparing the two closes that gap for every order type, not just the one
(GTC) that doesn't resolve synchronously in its own response. Hooking it into `live.rs` (subscribe
once at startup, `watch()` each order_id right after submission) also still delivers the original
functional upgrade: event-driven fill confirmation for the unwind leg instead of only discovering
the fill on the next poll cycle.

**E. A small always-on latency probe (larger effort, do only if A–D and F turn out insufficient).**
A dedicated lightweight Rust binary (or a mode of the existing collector) that, on a fixed
cadence (e.g. every 5s), issues a cheap authenticated read (e.g. `balance_allowance`) through the
same EC2 proxy path production orders use, times the round trip, and records it to a small
parquet/CSV alongside the existing feed-latency data. Unlike B (which only produces samples when
the bot actually trades), this gives a continuous baseline of "how much does the EC2 proxy hop
cost right now" even during quiet periods, useful for distinguishing "this specific order was
slow" from "the proxy itself was slow at that moment." Only worth building if B's per-trade
samples turn out too sparse or too noisy to trust (this system does relatively few live trades
per day, per the existing CSV logs).

> **Feedback (2026-07-04):** maybe later — leave as lowest-priority backlog item, don't schedule.

**F. Binance: match parquet cadence to CLOB (250ms → 200ms) and publish to NATS immediately
(new, from 2026-07-04 feedback — tentative, "not sure" per feedback, listed for a go/no-go
rather than committed).**

> **Feedback (2026-07-04) — Q: what's the current cadence for calculating spread?** Split by
> side, since `SpreadSignal` and `DeltaPctSignal` are driven by different events
> (`worker.rs::on_poly` / `on_binance`):
> - **Poly-side (`SpreadSignal::on_poly`)** recalculates on **every** `PolyTick`, and
>   `spawn_bba_task` (collect.rs:940-1051) already publishes every merged best-bid-ask/
>   price-change event to NATS immediately, uncapped by any ticker — so this is already
>   effectively real-time, single-digit-ms-latency per §3.1.
> - **Binance-side (`DeltaPctSignal::on_binance`)** only recalculates when a `BinanceTick`
>   arrives over NATS, and today `ticker_250ms` in `collect.rs::run` (collect.rs:1263-1284) is the
>   **only** place that publishes `price.binance.<ASSET>` — `spawn_binance_task` updates the
>   shared `BinanceState` on every raw trade, but that update just sits there until the next
>   250ms tick picks it up. So the delta-spread signal effectively recalculates at most every
>   ~250ms today, not per Binance trade — and per §6.3's measured gap table, 7.8–17.8% of
>   consecutive Binance trades (varies by asset) already land inside a single 250ms window, so
>   this throttle is provably discarding distinct trade prices, not just a theoretical ceiling.

> **Proposed change, pending go-ahead:** (1) change the Binance parquet sampler from
> `ticker_250ms` to the existing `ticker_200ms` (collect.rs:1233-1234) so the recorded Binance
> cadence lines up with the 200ms poly/book 5m+15m cadence instead of running its own 250ms
> clock — purely a recording-granularity change, same kind of thing the 200ms/1s split already
> is for poly. (2) Change `spawn_binance_task` (or wherever the Binance NATS publish happens) to
> publish to `price.binance.<ASSET>` **immediately on every WS trade event**, mirroring
> `spawn_bba_task`'s pattern for poly — decoupling the *live NATS/trading* path from the
> *parquet-recording* cadence entirely, same split that already exists for poly (§6.3 point 2 vs.
> point 1). Net effect: `on_binance`/`DeltaPctSignal` would recalculate on every real Binance
> trade instead of at most 4x/sec, removing the ~100-200ms class of added reaction lag identified
> in §3.2/§6.3 point 3 (on top of, not instead of, the ~115ms network floor, which is real
> network distance and not reducible this way).

**G. Persist DOWN-side book snapshots instead of dropping them (small, from 2026-07-04 feedback,
follows from §6.4).** `spawn_book_task` already receives the DN-token book message on the wire
(collect.rs:908-922) and currently discards it (`if is_up { ... }`, no `else`). Store it — extend
`AssetWriters`/`write_sample` with DN-side columns populated from the *actual* received DN book
message, alongside (not instead of) the existing derived `1 - UP` columns, so the "DN price is
always `1 - UP`" assumption running through the rest of this doc (§3.1, §6.4) can be checked
against real data rather than asserted from reading the code.

**Suggested order:** A and the timestamp part of B first (both are near-zero risk, no behavior
change, pure logging), then the rest of B, then C, then D (now broader per feedback), then G
(small, self-contained). F needs a go/no-go before scheduling (feedback was "not sure") — if

## 9. Implemented (2026-07-04): entry evaluation now triggers on poly ticks too

Follow-on from the discussion above §8 ("is the engine lazy until the price trigger is first
reached?"): poly price is the primary/time-critical signal for both `ReversalStrategy` and
`HighProbStrategy`; `delta_pct` is a directional filter. The live worker (`worker.rs`) and the
backtest engine (`machine.rs`) previously evaluated entries **only** on `BinanceTick` — a poly
price crossing its trigger band between Binance ticks sat unnoticed until the next Binance tick
happened to arrive (up to the Binance feed's own tick interval, currently ~250ms per §6.3).

**Change:** both `on_binance` and `on_poly` now call a shared `try_enter(now)` — entry can fire
off either feed, using the latest cached value of the other. `check_gates`'s existing
`|delta_pct| >= threshold` gate still applies regardless of which tick triggered evaluation, so
this doesn't loosen the actual entry condition, just how promptly it's checked. Kept `worker.rs`
(live) and `machine.rs` (backtest) in sync so backtest stays representative of live behavior.

**Correctness fix required alongside it:** `DeltaPctSignal::reset()` previously cleared `open`
but not `price` — harmless under the old on-Binance-only design (on_binance always refreshes
`price` in the same call that evaluates it), but a real bug once `on_poly` can trigger evaluation
without first refreshing `price` itself: a stale Binance price left over from the *previous*
cycle could otherwise pass as this cycle's already-known delta. Fixed by clearing `price` on
every `reset()`, same as `open`.

**Tests added:** `signal/delta_pct.rs::reset_clears_stale_price_from_previous_cycle`;
`worker.rs::entry_fires_on_poly_tick_using_cached_delta` and
`::poly_tick_does_not_fire_using_stale_cross_cycle_delta`; matching pair in `machine.rs`. Existing
tests that asserted entry firing on a specific final `BinanceTick` step were adjusted (the early,
redundant Binance tick that used to pre-seed `delta_pct` was removed so those tests still exercise
the "Binance is the last piece to arrive" path unchanged) — see `enter_down_position` and
`entry_fires_and_transitions_to_entering` in `worker.rs`.

**Real-data consequence, not just a synthetic-test change:** `backtest::btc_20260620_golden`
(replays real BTC parquet data validated against the original Python engine) now finds a 2nd
trade it didn't before, on cycle `btc-updown-5m-1781891400`. Traced it: poly `up` spiked
0.145 → 0.605 in under half a second while Binance ticks in that window landed roughly once per
second — the old design couldn't see the crossing in time; the new one catches it off the poly
tick itself using the already-cached Binance delta. Confirmed against the raw parquet ticks before
accepting this as correct and updating the golden test's expected values (now 2 trades). This is
the intended effect of the change, but flagging here because it's a real, quantified divergence
from the original Python-validated reference on this specific historical day, not just an
architectural refactor — worth keeping in mind if backtest P&L numbers are compared against older
Python-era runs.
