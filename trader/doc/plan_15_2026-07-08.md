# Plan â€” integrate 15-minute "updown" crypto markets as an independent product alongside 5m

Status: **draft for review, not yet implemented.** No code in this plan has been written.

## 0. Goal and constraint

Add a second, fully independent product trading Polymarket's 15-minute updown markets â€”
own config, own naming, own log/state files, own systemd service â€” running *alongside* the
existing 5-minute (`live_trades_eth_high_prob.csv`-style) product, never merged into it and never
sharing a process. The two must be able to run, fail, halt, and redeploy completely independently.

## 1. Where things already stand (research summary)

- **`price_feed` already records 15m data live, in production, unconditionally** â€”
  `collect.rs::run()` spawns a full second set of book/bba/trade/meta tasks at `900` seconds
  alongside the 5m (`300`) and 4hr (`14400`) ones (collect.rs:1228-1245), writing to
  `raw_15_mins/`/`raw_new_15_mins/`. These are **not** legacy directories â€” README.md's 2026-07-04
  integrity audit already treats them as live production output. **No price_feed changes needed**
  to collect 15m data, with one caveat below (Â§1.1).
- **The trading engine's core (state machine, signals, strategies, config loader) is already
  duration-agnostic.** `worker.rs`, `machine.rs`, `strategies.rs`, `signal/*.rs`, `config.rs`,
  `config_log.rs` take cycle length purely from `CycleContext.end_ts` / plain-second config values
  passed in by the caller â€” none of them hard-code `300`/`5m` in production code (only in
  `#[cfg(test)]` fixtures). This means most of the "port to 15m" work is **new config +
  new CLI invocation + a couple of targeted bug fixes**, not a rewrite.
- **`src/bin/live.rs` already has `--period-secs`** (default 300) correctly wired into cycle-boundary
  math, but the **slug suffix is hard-coded to `"5m"`** at the one call site that resolves which
  market to actually fetch/trade (`live.rs:1172`, `make_slug(&asset, slot_now, "5m")`). This is the
  single most important fix â€” see Â§1.1.
- **`backtest.rs` hard-codes `CYCLE_LENGTH_S = 300.0`** and `bin/backtest.rs` has no
  `--period-secs`/`--cycle-length-s` flag â€” needs parametrizing before a 15m backtest can run at
  all.
- **`deploy_oracle.py`/systemd are single-instance today** â€” one binary path (fine, shared), but one
  hard-coded config dir, log dir, and systemd unit name. A second, independent unit is needed.
- **A sibling Python bot (`../btc_5mins`, outside this repo) already solved the equivalent
  generalization** for its backtester (`--market {5m,15m,4h}`, proportional param scaling) and has
  real, if preliminary, 15m calibration study results â€” directly useful as a starting point for the
  new TOML's values (Â§4).

### 1.1 Critical bug this plan must fix before 15m can be run at all

`900` seconds is an exact multiple of `300`, so `current_slot(900)` always lands on a timestamp
that also happens to be a valid *5-minute* slot boundary. If `live --period-secs 900` were run
today without also fixing the slug suffix, `make_slug`/`fetch_meta` would **successfully resolve a
real 5-minute market's token IDs** (no error) while the rest of the code treated it as a 900-second
cycle â€” i.e. it would silently keep trading a 5-minute market's UP/DOWN tokens for 15 minutes after
that market already resolved and rotated. This is a silent-correctness bug, not a missing feature,
and is priority #1 in Â§2.

A closely related bug: price_feed's 15m Poly (CLOB) feed does **not** publish to NATS today â€”
only the 5m bba task does (`collect.rs:1231` passes the real NATS client; the 15m task at
`collect.rs:1239` passes `None`). If a 15m `live` instance were run with `--nats-url` as-is, it
would silently receive the **5m** market's UP/DOWN prices. Must be fixed or explicitly worked
around (Â§2.1) before NATS mode is usable for 15m.

## 2. Changes required, by area

### 2.1 `trader/src/bin/live.rs` â€” slug suffix + NATS feed selection

- Add a `--slug-suffix` CLI flag (default `"5m"`, matching current behavior exactly for existing
  5m deployments) and use it at the `make_slug(&asset, slot_now, "5m")` call site (live.rs:1172)
  instead of the literal. Mirrors the pattern `src/bin/shadow.rs` already uses correctly
  (`shadow.rs:150`, `--suffix` flag).
- Decide feed mode for the 15m instance:
  - **Option A (recommended to start): direct-WS mode** (omit `--nats-url`). `marketdata.rs`'s
    `fetch_meta`/`spawn_poly_task` are fully generic per-token/per-market, unaffected by the NATS
    gap. Simplest, no price_feed change needed, ships fastest.
  - **Option B: extend `collect.rs` to publish 15m Poly ticks under a distinct NATS subject**
    (e.g. `price.poly.15m.{asset}`), and have `live.rs` subscribe to the right subject based on its
    own `--slug-suffix`. More consistent with how 5m already runs in production (NATS mode), but
    is additional price_feed work and a second thing to get right before going live.
  - Recommendation: ship with Option A first: prove out the 15m product in direct-WS mode, then
    move to Option B once it's stable, matching how 5m itself presumably started.
- `src/bin/api_probe.rs` has the same `"5m"`/`300` hard-coding (api_probe.rs:115-116) â€” low
  priority, it's a manual diagnostic probe, not part of the live path; fix opportunistically.

### 2.2 `trader/src/backtest.rs` + `trader/src/bin/backtest.rs`

- Turn `CYCLE_LENGTH_S: f64 = 300.0` (backtest.rs:220) into a parameter threaded through
  `run_backtest`/`replay_cycle` rather than a module constant.
- Add `--period-secs` (or `--cycle-length-s`) to `bin/backtest.rs`'s `Args`, default `300` to
  preserve current behavior for existing 5m backtests.
- `trader/scripts/build_backtest_prices.py` is hard-coded to `price_feed/raw`
  (`RAW_DIR`, build_backtest_prices.py:43) â€” add a `--market`/`--raw-dir` flag so it can point at
  `price_feed/raw_15_mins` instead, mirroring the sibling Python bot's `scripts/bt1.py --market`.
- This unblocks running a real Rust backtest against `raw_15_mins` history to validate whatever
  config values Â§4 proposes, before any live capital is at risk.

### 2.3 Config â€” new, wholly separate config directory (not a schema change)

`config.rs`/`config_log.rs` already take `--config-dir`/`--log-dir` as plain CLI args with no
duration coupling â€” this means a 15m product needs **zero `config.rs` code changes**, just:

- A new directory, e.g. `trader/config_15m/strategy_15m_YYYYMMDD.toml`, following the exact same
  schema as today's `strategy_20260705.toml` (`trade_assets`, per-strategy param tables, etc.), with
  its own values (Â§4) â€” **not** a shared file with a `[BTC.15m]`-style nested key. `plan_rust_module.md`
  Â§13 originally proposed the nested-key approach; rejecting it here in favor of two fully separate
  files/directories, since the two products must never share a process anyway and a schema change
  would touch `StrategyToml`'s per-asset maps for no isolation benefit.
- A new `--log-dir live_logs_15m` for the 15m `live` invocation. File naming in `live.rs`
  (`live_trades_{asset}_{strategy}.csv`, `live_state_{asset}_{strategy}.json`) has **no duration
  component and is already collision-free** as long as `--log-dir` differs â€” confirmed no code
  change needed here.

### 2.4 Deploy / systemd (`scripts/deploy_oracle.py`, `scripts/deploy_trader.sh`)

Currently single-instance by hard-coded module constants (`TRADER_CFG_DIR`, `TRADER_LOG_DIR`,
`TRADER_SERVICE = "trader-live.service"`, `TRADER_UNIT_PATH`, `TRADER_ASSETS`). Needed:

- A parallel set of constants (or a parametrized `_trader_unit_file(product: str, ...)`) for:
  new Oracle config dir (e.g. `.../btc_5mins/config_15m`), new log dir

  `TRADER_ASSETS_15M` sourced from the new config dir's own latest `strategy_15m_*.toml`.
- The rendered `ExecStart` for the new unit must include `--period-secs 900 --slug-suffix 15m`
  (once Â§2.1 ships) alongside its own `--config-dir`/`--log-dir`.
- A `--product {5m,15m}` flag (or a second `deploy_trader_15m` function) threaded through
  `deploy_oracle.py`'s CLI and `deploy_trader.sh`.
- **Must keep the existing `systemctl restart <unit>`-only discipline** (never manual

### 2.5 Telegram, `execution.rs`, `unwind.rs`, `redemption.rs`, `gates.rs`, `balance.rs`

No changes needed â€” confirmed fully generic (Telegram renders whatever asset/strategy/slug string

## 4. Where to source initial 15m strategy parameters

`enter_when_time_left`/`no_enter_when_time_left`/`reversal_start_time`/`unwind_time_*` in the
current 5m TOML are absolute-second values tuned for a 300s cycle â€” they'll *run* unmodified
against a 900s cycle (the code doesn't care), but represent very different fractions of the cycle
(e.g. `reversal_start_time=120` is 40% of a 300s cycle, 13% of a 900s one), so they need their own
15m-specific calibration, not a shared file.

Two existing sources to seed from, both in the sibling `../btc_5mins` repo (outside this repo, not
requested to touch, but directly relevant as reference data):

- `bot/backtest.py`/`bt1.py` already implement `_scaled_secs(v, cycle_length_s) = round(v *
  cycle_length_s / 300.0)` â€” a proportional-scaling model for these window parameters.
- `studies/15_mins/` and `studies/bt2/summary_reversal_with_unwind_2026-07-04.md` already ran real
  (if data-limited: ~19 clean 15m days vs. 5m's ~47) backtests and found `high_prob` performs well
  (~100% win rate, positive PnL, all 6 assets) with best-fit `enter_when_time_left` around **T-20 to
  T-25 â€” notably close to 5m's own default, not proportionally scaled Ă—3**, which is in tension
  with the `_scaled_secs` assumption above. `reversal` fires rarely at 15m (0-5 trades in the
  studied window) and is flagged "inconclusive."

**Recommendation:** don't blindly proportional-scale the 5m TOML. Start from the `studies/15_mins`
best-fit values directly, re-run them through the now-parametrized Rust backtest (Â§2.2) against the
larger `raw_15_mins` history that's accumulated since 2026-06-13, and treat `reversal` at 15m as
"disabled until proven" (`ETH = ["high_prob"]` only, no `reversal`, until a proper walk-forward
shows it's worth including) rather than shipping both strategies day one the way 5m does.

## 8. Claude thoughts on the DeepSeek review

Spot-checked each DeepSeek finding against the current code before trusting it. Net: **findings
1/9 (shared-account balance contamination) and 5 (Telegram token collision) are real, verified,
and materially change what "ship it" requires â€” they should be treated as blocking, not just
open questions.** Finding 4 doesn't hold up. The rest are reasonable but lower-stakes process
suggestions.

- **#1/#9 (shared-account `BalanceGuard`) â€” confirmed, and worse than DeepSeek's framing.**
  Read `trader/src/balance.rs`: `BalanceGuard` tracks one `initial_balance` baseline set at
  process start and halts on >25% drawdown from that baseline (`DRAWDOWN_LIMIT`, `check()` at
  balance.rs:45-61) â€” it has no concept of "which product caused this drawdown," it just polls
  the account's total USDC balance once a cycle. Two processes against one account means the 15m
  instance's losses *will* show up in the 5m instance's drawdown check and vice versa. This isn't
  a tail-risk edge case, it's the expected steady-state behavior of Â§0's "own everything,
  independent process" design applied to something that explicitly *isn't* independent â€” the
  wallet. **This should block go-live, not just be a documented tradeoff.** Either get a second
  API key/sub-account before deploying 15m live capital, or explicitly size `DRAWDOWN_LIMIT`/
  `--size-usdc` so that a full loss of one product's max exposure can't trip the other's guard,
  and say so in Â§2.5 (which currently asserts the opposite â€” "independent `BalanceGuard`... fine,
  requires no fix").

- **#5 (Telegram) â€” confirmed, and there's a more specific failure mode than DeepSeek stated.**
  `live.rs`'s own file-header comment (lines 4-9) already documents this exact hazard for the
  *existing* multi-asset design: "running one process per asset means N processes each try to
  long-poll Telegram's `getUpdates` on the same bot token, and only one wins (the others get
  silent 409 Conflicts), breaking remote control for everyone but the first-started process."
  DeepSeek framed this as "both processes see the same command," which is the opposite of what
  actually happens â€” the polling race means only **one** of the two processes' Telegram loop
  wins, and the other silently stops receiving `/halt`, `/resume`, or any other remote command
  at all, with no error surfaced anywhere. Nothing in `config.rs` allows overriding
  `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` per-instance â€” it only comes from `--env-file`
  (default shared: `/home/kev/apps/btc_5mins/.env`), and the plan's Â§2.3 doesn't mention
  `--env-file` at all. **This needs an explicit line item**: either a distinct `--env-file` with
  its own `TELEGRAM_BOT_TOKEN` for the 15m instance, or an explicit decision to accept that
  whichever instance starts second has no working remote control â€” the current plan is silent on
  this and would ship with silently broken `/halt` on one product.

- **#4 (backtest day-boundary partitioning) â€” checked, doesn't hold up.** DeepSeek's concern was
  that `backtest.rs` might use `(end_ts - start_ts) / CYCLE_LENGTH_S`-style integer division to
  partition raw ticks into cycles, which could produce partial/misaligned cycles at day
  boundaries for a 900s period. Read `run_backtest`/`replay_cycle` (backtest.rs:458-521): cycles
  are grouped by the row's own `slug` field into a `HashMap<String, Vec<_>>` (`b_by_slug`/
  `p_by_slug`), not by dividing timestamps â€” `CYCLE_LENGTH_S` is used in exactly one place
  (`cycle_end_ts = slug_ts + CYCLE_LENGTH_S`, backtest.rs:266) to build the `CycleContext` handed
  to the state machine, not to bucket ticks. So there's no day-boundary partitioning bug. The real
  risk hiding under this finding is simpler and already correctly identified by Â§2.2 itself: until
  `CYCLE_LENGTH_S` is parametrized, `replay_cycle` would silently hand the strategy machine a
  5-minute `end_ts` while replaying 15 minutes of actual tick data (since raw_15_mins ticks
  already carry correct 15m slugs per `price_feed`'s `spawn_meta_task(..., 900, "15m")`), which
  would corrupt every time-left-dependent signal without erroring. Good reason to keep Â§2.2's
  parametrization on the critical path before any 15m backtest number is trusted, but not a new
  finding beyond what Â§2.2 already says.

- **#2, #3, #6, #7, #8, #10 â€” reasonable, non-blocking.** #3's suggestion (derive slug-suffix
  from `--period-secs` instead of two independent flags, e.g. `300 -> "5m"`, `900 -> "15m"`) is a
  cheap, good change worth folding into Â§2.1 directly rather than leaving as an open question â€”
  it closes the exact misconfiguration path Â§1.1 is worried about. #7 (don't bundle the
  `live.rs` fix with `deploy_oracle.py` changes) matches how Â§5 is already ordered â€” worth calling
  out explicitly as a constraint rather than just an implication of the list order. #10 is a
  correction in the plan's favor: DeepSeek confirmed the plan's Â§4 conservative-default
  reasoning is *reinforced* by the newer 5m reversal-only-by-default precedent, not undercut by
  it â€” worth citing when Â§4 is finalized. #2, #6, #8 are worth a line each in Â§2.1/Â§2.3/Â§2.4 but
  don't change the plan's structure.
