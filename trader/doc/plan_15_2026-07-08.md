# Plan — integrate 15-minute "updown" crypto markets as an independent product alongside 5m

Status: **draft for review, not yet implemented.** No code in this plan has been written.

## 0. Goal and constraint

Add a second, fully independent product trading Polymarket's 15-minute updown markets —
own config, own naming, own log/state files, own systemd service — running *alongside* the
existing 5-minute (`live_trades_eth_high_prob.csv`-style) product, never merged into it and never
sharing a process. The two must be able to run, fail, halt, and redeploy completely independently.

## 1. Where things already stand (research summary)

- **`price_feed` already records 15m data live, in production, unconditionally** —
  `collect.rs::run()` spawns a full second set of book/bba/trade/meta tasks at `900` seconds
  alongside the 5m (`300`) and 4hr (`14400`) ones (collect.rs:1228-1245), writing to
  `raw_15_mins/`/`raw_new_15_mins/`. These are **not** legacy directories — README.md's 2026-07-04
  integrity audit already treats them as live production output. **No price_feed changes needed**
  to collect 15m data, with one caveat below (§1.1).
- **The trading engine's core (state machine, signals, strategies, config loader) is already
  duration-agnostic.** `worker.rs`, `machine.rs`, `strategies.rs`, `signal/*.rs`, `config.rs`,
  `config_log.rs` take cycle length purely from `CycleContext.end_ts` / plain-second config values
  passed in by the caller — none of them hard-code `300`/`5m` in production code (only in
  `#[cfg(test)]` fixtures). This means most of the "port to 15m" work is **new config +
  new CLI invocation + a couple of targeted bug fixes**, not a rewrite.
- **`src/bin/live.rs` already has `--period-secs`** (default 300) correctly wired into cycle-boundary
  math, but the **slug suffix is hard-coded to `"5m"`** at the one call site that resolves which
  market to actually fetch/trade (`live.rs:1172`, `make_slug(&asset, slot_now, "5m")`). This is the
  single most important fix — see §1.1.
- **`backtest.rs` hard-codes `CYCLE_LENGTH_S = 300.0`** and `bin/backtest.rs` has no
  `--period-secs`/`--cycle-length-s` flag — needs parametrizing before a 15m backtest can run at
  all.
- **`deploy_oracle.py`/systemd are single-instance today** — one binary path (fine, shared), but one
  hard-coded config dir, log dir, and systemd unit name. A second, independent unit is needed.
- **A sibling Python bot (`../btc_5mins`, outside this repo) already solved the equivalent
  generalization** for its backtester (`--market {5m,15m,4h}`, proportional param scaling) and has
  real, if preliminary, 15m calibration study results — directly useful as a starting point for the
  new TOML's values (§4).

### 1.1 Critical bug this plan must fix before 15m can be run at all

`900` seconds is an exact multiple of `300`, so `current_slot(900)` always lands on a timestamp
that also happens to be a valid *5-minute* slot boundary. If `live --period-secs 900` were run
today without also fixing the slug suffix, `make_slug`/`fetch_meta` would **successfully resolve a
real 5-minute market's token IDs** (no error) while the rest of the code treated it as a 900-second
cycle — i.e. it would silently keep trading a 5-minute market's UP/DOWN tokens for 15 minutes after
that market already resolved and rotated. This is a silent-correctness bug, not a missing feature,
and is priority #1 in §2.

A closely related bug: price_feed's 15m Poly (CLOB) feed does **not** publish to NATS today —
only the 5m bba task does (`collect.rs:1231` passes the real NATS client; the 15m task at
`collect.rs:1239` passes `None`). If a 15m `live` instance were run with `--nats-url` as-is, it
would silently receive the **5m** market's UP/DOWN prices. Must be fixed or explicitly worked
around (§2.1) before NATS mode is usable for 15m.

## 2. Changes required, by area

### 2.1 `trader/src/bin/live.rs` — slug suffix + NATS feed selection

- Add a `--slug-suffix` CLI flag (default `"5m"`, matching current behavior exactly for existing
  5m deployments) and use it at the `make_slug(&asset, slot_now, "5m")` call site (live.rs:1172)
  instead of the literal. Mirrors the pattern `src/bin/shadow.rs` already uses correctly
  (`shadow.rs:150`, `--suffix` flag).
- Decide feed mode for the 15m instance:
  - **Option A (recommended to start): direct-WS mode** (omit `--nats-url`). `marketdata.rs`'s
    `fetch_meta`/`spawn_poly_task` are fully generic per-token/per-market, unaffected by the NATS
    gap. Simplest, no price_feed change needed, ships fastest.
  - **Option B: extend `collect.rs` to publish 15m Poly ticks under a distinct NATS subject**
    (e.g. `price.poly.15m.{asset}`), and have `live.rs` subscribe to the right subject based on its
    own `--slug-suffix`. More consistent with how 5m already runs in production (NATS mode), but
    is additional price_feed work and a second thing to get right before going live.
  - Recommendation: ship with Option A first: prove out the 15m product in direct-WS mode, then
    move to Option B once it's stable, matching how 5m itself presumably started.
- `src/bin/api_probe.rs` has the same `"5m"`/`300` hard-coding (api_probe.rs:115-116) — low
  priority, it's a manual diagnostic probe, not part of the live path; fix opportunistically.

### 2.2 `trader/src/backtest.rs` + `trader/src/bin/backtest.rs`

- Turn `CYCLE_LENGTH_S: f64 = 300.0` (backtest.rs:220) into a parameter threaded through
  `run_backtest`/`replay_cycle` rather than a module constant.
- Add `--period-secs` (or `--cycle-length-s`) to `bin/backtest.rs`'s `Args`, default `300` to
  preserve current behavior for existing 5m backtests.
- `trader/scripts/build_backtest_prices.py` is hard-coded to `price_feed/raw`
  (`RAW_DIR`, build_backtest_prices.py:43) — add a `--market`/`--raw-dir` flag so it can point at
  `price_feed/raw_15_mins` instead, mirroring the sibling Python bot's `scripts/bt1.py --market`.
- This unblocks running a real Rust backtest against `raw_15_mins` history to validate whatever
  config values §4 proposes, before any live capital is at risk.

### 2.3 Config — new, wholly separate config directory (not a schema change)

`config.rs`/`config_log.rs` already take `--config-dir`/`--log-dir` as plain CLI args with no
duration coupling — this means a 15m product needs **zero `config.rs` code changes**, just:

- A new directory, e.g. `trader/config_15m/strategy_15m_YYYYMMDD.toml`, following the exact same
  schema as today's `strategy_20260705.toml` (`trade_assets`, per-strategy param tables, etc.), with
  its own values (§4) — **not** a shared file with a `[BTC.15m]`-style nested key. `plan_rust_module.md`
  §13 originally proposed the nested-key approach; rejecting it here in favor of two fully separate
  files/directories, since the two products must never share a process anyway and a schema change
  would touch `StrategyToml`'s per-asset maps for no isolation benefit.
- A new `--log-dir live_logs_15m` for the 15m `live` invocation. File naming in `live.rs`
  (`live_trades_{asset}_{strategy}.csv`, `live_state_{asset}_{strategy}.json`) has **no duration
  component and is already collision-free** as long as `--log-dir` differs — confirmed no code
  change needed here.

### 2.4 Deploy / systemd (`scripts/deploy_oracle.py`, `scripts/deploy_trader.sh`)

Currently single-instance by hard-coded module constants (`TRADER_CFG_DIR`, `TRADER_LOG_DIR`,
`TRADER_SERVICE = "trader-live.service"`, `TRADER_UNIT_PATH`, `TRADER_ASSETS`). Needed:

- A parallel set of constants (or a parametrized `_trader_unit_file(product: str, ...)`) for:
  new Oracle config dir (e.g. `.../btc_5mins/config_15m`), new log dir
  (`{ORACLE_BASE}/trader/live_logs_15m`), new systemd unit name/path
  (`trader-live-15m.service` / `/etc/systemd/system/trader-live-15m.service`), and
  `TRADER_ASSETS_15M` sourced from the new config dir's own latest `strategy_15m_*.toml`.
- The rendered `ExecStart` for the new unit must include `--period-secs 900 --slug-suffix 15m`
  (once §2.1 ships) alongside its own `--config-dir`/`--log-dir`.
- A `--product {5m,15m}` flag (or a second `deploy_trader_15m` function) threaded through
  `deploy_oracle.py`'s CLI and `deploy_trader.sh`.
- **Must keep the existing `systemctl restart <unit>`-only discipline** (never manual
  `kill`/`tmux`) for the new unit too — this codebase already has a documented incident
  (2026-07-03, deploy_oracle.py:17-29) from exactly this kind of double-process race; a second,
  independent unit doubles the surface for that class of mistake if deploy tooling isn't equally
  disciplined for it.

### 2.5 Telegram, `execution.rs`, `unwind.rs`, `redemption.rs`, `gates.rs`, `balance.rs`

No changes needed — confirmed fully generic (Telegram renders whatever asset/strategy/slug string
it's given; the order/exit/redemption/gate modules have no duration or slug coupling at all).
`balance.rs`'s `WINDOW_SECS=300` poll cadence is unrelated to market cycle length (it's a
USDC-balance drawdown check, not a market timer) — a 15m instance runs its own independent
`BalanceGuard` on the same cadence, which is fine and requires no fix.

## 3. Naming convention for new artifacts

| Artifact | 5m (existing) | 15m (new) |
|---|---|---|
| Slug suffix | `5m` (`eth-updown-5m-<epoch>`) | `15m` (`eth-updown-15m-<epoch>`) |
| price_feed dir | `raw/`, `raw_new/` | `raw_15_mins/`, `raw_new_15_mins/` (already exist) |
| Trader config dir | `trader/config/strategy_*.toml` | `trader/config_15m/strategy_15m_*.toml` |
| Trader log dir | `trader/live_logs/` | `trader/live_logs_15m/` |
| CSV/state files | `live_trades_eth_high_prob.csv` etc. | same pattern, under `live_logs_15m/` — no name change needed |
| systemd unit | `trader-live.service` | `trader-live-15m.service` |
| CLI flags on `live` | `--period-secs 300` (default), no suffix flag | `--period-secs 900 --slug-suffix 15m` |

## 4. Where to source initial 15m strategy parameters

`enter_when_time_left`/`no_enter_when_time_left`/`reversal_start_time`/`unwind_time_*` in the
current 5m TOML are absolute-second values tuned for a 300s cycle — they'll *run* unmodified
against a 900s cycle (the code doesn't care), but represent very different fractions of the cycle
(e.g. `reversal_start_time=120` is 40% of a 300s cycle, 13% of a 900s one), so they need their own
15m-specific calibration, not a shared file.

Two existing sources to seed from, both in the sibling `../btc_5mins` repo (outside this repo, not
requested to touch, but directly relevant as reference data):

- `bot/backtest.py`/`bt1.py` already implement `_scaled_secs(v, cycle_length_s) = round(v *
  cycle_length_s / 300.0)` — a proportional-scaling model for these window parameters.
- `studies/15_mins/` and `studies/bt2/summary_reversal_with_unwind_2026-07-04.md` already ran real
  (if data-limited: ~19 clean 15m days vs. 5m's ~47) backtests and found `high_prob` performs well
  (~100% win rate, positive PnL, all 6 assets) with best-fit `enter_when_time_left` around **T-20 to
  T-25 — notably close to 5m's own default, not proportionally scaled ×3**, which is in tension
  with the `_scaled_secs` assumption above. `reversal` fires rarely at 15m (0-5 trades in the
  studied window) and is flagged "inconclusive."

**Recommendation:** don't blindly proportional-scale the 5m TOML. Start from the `studies/15_mins`
best-fit values directly, re-run them through the now-parametrized Rust backtest (§2.2) against the
larger `raw_15_mins` history that's accumulated since 2026-06-13, and treat `reversal` at 15m as
"disabled until proven" (`ETH = ["high_prob"]` only, no `reversal`, until a proper walk-forward
shows it's worth including) rather than shipping both strategies day one the way 5m does.

## 5. Suggested implementation order

1. `live.rs`: add `--slug-suffix`, fix the hard-coded call site (§2.1) — ship and manually smoke-test
   in direct-WS mode against a real 15m market slug, no capital at risk (dry-run/paper mode if one
   exists, else a `--size-usdc` near-zero test).
2. `backtest.rs`/`bin/backtest.rs`/`build_backtest_prices.py`: parametrize cycle length (§2.2).
3. Run the parametrized Rust backtest against `raw_15_mins` using `studies/15_mins`-derived starting
   parameters (§4); compare against the Python study's own findings as a cross-check.
4. Write `trader/config_15m/strategy_15m_<date>.toml` from the validated backtest run.
5. `deploy_oracle.py`/`deploy_trader.sh`: add the parallel `trader-live-15m.service` path (§2.4).
6. Deploy to Oracle with `trade_assets` scoped to a single low-risk asset first (matching how 5m
   itself is currently scoped to `ETH` only post-2026-07-07), `high_prob` only, small size, and
   monitor for at least one full multi-day window before widening scope.

## 6. Open questions for review

- Confirm Option A vs. B for the NATS feed gap (§2.1) — direct-WS is simpler to ship but 5m's
  production path uses NATS; worth knowing if there's a reason NATS was chosen for 5m that also
  applies here (e.g. running on a machine where opening extra WS connections is undesirable).
- Confirm the `reversal`-disabled-at-15m recommendation (§4) is acceptable, or whether you'd rather
  ship both strategies from day one and let the halt/loss-streak mechanism contain a bad `reversal`
  config.
- Confirm asset scope for the initial 15m rollout (single asset like 5m's current `ETH`-only scope,
  or broader) — this plan defaults to assuming "start narrow," matching the 5m precedent.
